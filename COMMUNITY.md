# AURA Community Draft

> Status: Draft. This is a proposed community and governance note for review.

## Community Shape

AURA should welcome people who care about local-first AI tools, multimodal interaction, browser automation, mobile-to-desktop workflows, and open device coordination.

Potential contributor paths:

- [ ] AI site adapter maintainers.
- [ ] Windows desktop/runtime contributors.
- [ ] Android companion contributors.
- [ ] Local network and protocol contributors.
- [ ] Documentation and onboarding contributors.
- [ ] Designers and workflow researchers.

## Decision Principles

- [ ] Prefer changes that make the tool more reliable for real users.
- [ ] Prefer open protocols and inspectable behavior.
- [ ] Keep the core small enough to understand.
- [ ] Let integrations grow through adapters and plugins.
- [ ] Treat privacy and local control as product features.

## Proposed Governance

- [ ] Maintainers review project direction, releases, and security-sensitive changes.
- [ ] Contributors can own specific adapters or modules over time.
- [ ] Large design changes should start as discussion documents in `docs/`.
- [ ] Breaking protocol changes should include migration notes.
- [ ] Release notes should distinguish finished features from experiments.

## Open Questions

- [ ] Should AURA publish formal RFCs for protocol changes?
- [ ] Should adapters have separate maintainers?
- [ ] Should mobile and desktop release cycles be independent?
- [ ] Should packaged releases live in Git LFS, GitHub Releases, or both?
- [ ] What security review is needed for plugins?
