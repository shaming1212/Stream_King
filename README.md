# AURA Voice Assistant

本地 AI 语音助手 + Chrome 浏览器插件联动系统。按住快捷键说话，本地 FunASR 引擎实时识别，通过 WebSocket + Chrome 扩展将文字/图片注入 Kimi、豆包、ChatGPT、DeepSeek、通义千问等 AI 对话页面。

## 快速启动

```bash
pip install -r requirements.txt
python main.py
```

首次启动会自动下载 AI 模型文件（约 500MB），请耐心等待。

## 快捷键

| 快捷键 | 功能 |
|--------|------|
| `Alt + 1` | 按住说话，松开识别并注入文字 |
| `Alt + 2` | 拍摄摄像头画面并注入图片 |
| `Alt + 3` | 屏幕截图选区并注入图片 |

## 安装 Chrome 扩展

1. 打开 Chrome，地址栏输入 `chrome://extensions/`
2. 开启右上角「开发者模式」
3. 点击「加载已解压的扩展程序」
4. 选择项目中的 `extension/` 文件夹
5. 确认扩展已启用，打开任意支持的 AI 对话页面即可使用

## 常见问题

### 启动后一直显示 LOADING？
首次运行需从 ModelScope 下载 AI 模型（约 500MB），网络速度影响较大，需等待 2-10 分钟。后续启动秒级就绪。

### 说话后文字没出现在网页中？
1. 确认 Chrome 扩展已启用
2. 确认 AI 对话页面是当前激活的标签页
3. 尝试刷新 AI 对话页面
4. 检查 WebSocket 连接状态（扩展 Service Worker 控制台可查看日志）

### 如何修改快捷键？
编辑 `core/config.py`，修改 `HOTKEY_VOICE`、`HOTKEY_CAMERA`、`HOTKEY_SCREENSHOT` 即可。快捷键格式参考 [keyboard 模块文档](https://github.com/boppreh/keyboard)。

### 端口 8765 被占用？
编辑 `core/config.py` 修改 `WS_PORT`，同时修改 `extension/background.js` 中的 `WS_URL`，保持一致。

## 愿景：全设备本地推流网络

> 让每一块屏幕都看见彼此，让每一次表达都穿透设备的边界。

### 我们在做什么

你身边的设备，从未真正连接在一起。

手机里一闪而过的灵感、平板上随手勾勒的草图、桌面上正在调试的终端日志——它们各自沉默，互不相知，被物理的边框钉死在各自的孤岛上。数据在设备之间流转，本应像空气一样自然。但现实是，你想把手机上的一张图弄到桌面上，还得经过微信、邮箱或者一根线。

AURA 想做的，就是在这片割裂的日常里，拉一根看不见的线。不止是语音助手——我们要把它打造成**本地网络的感知中枢**，一个运行在你主力机上的全设备推流服务平台。它静默地守在 localhost 背后，等你的每一台设备回家。

### 它长什么样

```
你坐在书房，桌面的 AURA 像一座安静的灯塔。

手机用 Safari 打开 192.168.1.x:8765 —— 无需安装任何 App。
平板靠过来，也连上了。
笔记本、客厅的树莓派、角落里的旧安卓机，都连上了。

你按下手机上的「推流镜头」，后置摄像头画面实时出现在桌面 Dashboard 上。
你用平板圈了一截设计稿，截图推流，桌面 AI 即时给出 4 条修改建议。
你对着手机说了一句话，它被路由到桌面 GPU 上跑的 ASR 模型，识别结果
  同时写进了 Kimi 输入框、终端日志、和客厅电视上的字幕条。
桌面正在编译的项目日志推流出去，你躺在卧室拿手机就能看进度。
树莓派挂载的摄像头推流到局域网，任何设备都能订阅——
  像订阅一个播客那样简单。

设备不再是孤岛。它们共享同一层感知、同一个 AI、同一种节奏。
```

### 功能蓝图

#### 第一阶段 · 推流基石（当前）

在现有 AURA 基础上，将单设备工具链升级为**可寻址的推流服务**：

| 模块 | 说明 |
|------|------|
| 流注册中心 | 每路流拥有唯一 `stream_id`，支持元数据描述（类型、分辨率、编码、设备名） |
| 频道模型 | 支持 `push`（设备上行）、`pull`（设备订阅）、`pub/sub`（多方广播）三种模式 |
| Web 控制台 | 浏览器端 Dashboard：设备拓扑图、流列表、实时预览缩略图、带宽仪表盘 |
| 设备发现 | mDNS / UDP 广播自动发现局域网内的 AURA 节点，扫码即连 |
| 安全边界 | Token 认证 + 设备白名单 + 可选的 TLS 自签名证书，陌生人无法推流 |

#### 第二阶段 · 全媒体流引擎

从文本/图片推流升级为**实时音视频流**：

| 模块 | 说明 |
|------|------|
| 音频流引擎 | 麦克风采集 → Opus 编码 → WebRTC/WebSocket 推流 → 多端同步播放 |
| 视频流引擎 | 摄像头/屏幕采集 → H.264/H.265 GPU 硬编码 → 自适应码率 → 低延迟分发 |
| 屏幕流引擎 | 全屏/窗口/选区三模式，支持鼠标高亮、点击动画叠加，帧率可调 5–60fps |
| WebRTC 信令服务 | NAT 穿透、STUN/TURN、SDP 协商，局域网内直连，外网自动中继 |
| 录制与回放 | 任意流可一键录制为 MP4/WebM，带时间轴回放，支持片段裁剪导出 |
| 多流合屏 | 画中画、画廊视图、自定义布局——多路流拼成一幅画面推送到指定频道 |

#### 第三阶段 · AI 处理管线

每一路流经过本地 AI 实时处理后再分发：

| 模块 | 说明 |
|------|------|
| 实时 ASR 管线 | 音频流 → SenseVoiceSmall 语音识别 → 实时字幕流，延迟 < 500ms |
| 实时 OCR 管线 | 屏幕/视频流 → PaddleOCR 文字提取 → 结构化文本流 |
| 视觉理解管线 | 视频帧 → 本地 VL 模型（如 Qwen-VL）→ 场景描述流 |
| 翻译管线 | 任意文本流 → 本地翻译模型 → 多语言字幕并行输出 |
| 处理链编排 | 拖拽式 DAG 编辑器：输入流 → 模型A → 模型B → 输出流，零代码搭建 AI 管道 |
| 结果回灌 | AI 处理结果可推回任意设备的任意输入框，形成"感知→理解→行动"闭环 |

#### 第四阶段 · 跨生态融合

| 模块 | 说明 |
|------|------|
| OBS 插件 | 一键将 AURA 频道导出为 OBS 虚拟摄像头/音频源，无缝接入直播工作流 |
| VRChat / 虚拟 avatar 驱动 | 将本地摄像头流接入 VMC 协议，驱动 3D 模型面部捕捉 |
| Home Assistant 集成 | 流事件触发自动化：检测到手机推流 → 自动点亮书桌灯、调高屏幕亮度 |
| 远程开发桥 | 将笔记本终端流推送到桌面，配合 AI 管线实现"手机写代码，桌面跑编译" |
| CLI 工具链 | `aura stream push`、`aura stream pull`、`aura stream record`，脚本友好 |
| SDK & 开放 API | Python / JavaScript / Rust SDK，RESTful API + WebSocket 双通道，社区可自由开发下游应用 |

### 估时

以下为单人全职开发预估，按每周 40 小时有效编码时间计算：

| 阶段 | 核心模块 | 预估工时 |
|------|----------|----------|
| **第一阶段** | 流注册中心 + 频道模型 + Web 控制台 + 设备发现 + 安全 | **6–8 周** |
| **第二阶段** | 音视频引擎（采集/编码/传输）+ WebRTC 信令 + 录制回放 + 合屏 | **12–16 周** |
| **第三阶段** | ASR/OCR/VL 管线 + 处理链编排 + 结果回灌 | **8–12 周** |
| **第四阶段** | OBS/VRChat/HA 集成 + 远程开发桥 + CLI + SDK | **8–10 周** |
| 测试 & 打磨 | 多设备兼容性测试、弱网压测、文档、示例项目 | **6–8 周** |
| **合计** | | **40–54 周** |

关键风险与复杂度：

| 风险点 | 说明 | 对策 |
|--------|------|------|
| WebRTC 跨平台兼容 | iOS Safari、Android Chrome、桌面各浏览器的 WebRTC 实现差异大 | 早期建立设备矩阵，每个迭代做全平台回归 |
| GPU 硬编码适配 | Windows/Linux/macOS 的编码栈完全不同（NVENC/VAAPI/VideoToolbox） | 优先支持软件编码回退，硬编码按平台逐个攻克 |
| 低延迟与画质平衡 | 局域网内 <100ms 延迟 + 可接受画质需要精细的参数调优 | 建立 benchmark 套件，量化每个改进的效果 |
| 多设备时钟同步 | 分布式设备间的音视频同步误差需 <50ms | NTP + PTP 混合方案，或基于音频指纹的自对齐 |
| AI 模型资源竞争 | GPU 同时跑 ASR + OCR + VL 管线可能导致推流丢帧 | 模型调度器 + 处理优先级队列 + 可选 CPU 回退 |

### 为什么值得做

我们活在一个设备比人多的时代。每个像素都困在自己的玻璃牢笼里。

把摄像头画面从手机搬到电脑，不该经过一个远在千里之外的服务器。你手机到桌面不到一米——信号本可以在空气里走完这段路。AURA 想成为那个让信号直接走完一米的人。不是发明新的协议，不是造轮子，而是把 WebRTC、WebSocket、mDNS 这些已经成熟的技术编织成一张每个人都能用的网，让你的每一块屏幕终于学会互相呼吸。

这不仅是一个推流工具。它是创作者的工作台、是开发者的调试室、是家庭实验室的中控台、是 VTuber 的推流机、是你在局域网里搭建的第一个属于自己的基础设施。

欢迎加入，一起搭建属于所有人的本地推流网络。
