# AURA Vision Draft

> Status: Draft. This document describes an unfinished direction for review.

AURA starts as a local voice assistant, but the larger ambition is wider: a local-first multimodal layer that lets people move intent, context, media, and commands across their own devices without turning every interaction into a cloud dependency.

## North Star

AURA should become a personal interaction layer for the AI age.

Not another chatbot window. Not another isolated desktop utility. AURA should be the bridge between speech, screens, cameras, phones, browsers, local models, and AI agents.

The long-term idea is simple:

- [ ] Speech should be a first-class interface.
- [ ] Screenshots and camera frames should be first-class context.
- [ ] Local devices should discover and help each other.
- [ ] AI tools should be composable instead of trapped in separate tabs.
- [ ] Users should own the path between their input and their AI workflow.

## Why This Matters

AI interfaces are still too narrow. A user speaks into one app, captures an image in another app, opens a browser tab somewhere else, and manually copies everything between them. The work is fragmented even when the intelligence is powerful.

AURA can become the missing connective tissue:

- [ ] Local speech to AI pages.
- [ ] Mobile capture to desktop AI workflows.
- [ ] Screen context to browser assistants.
- [ ] Local network discovery instead of account-bound cloud routing.
- [ ] Open adapters for new AI surfaces.

## Great Open Source Ambition

The project can grow into an open local AI operating layer:

- [ ] A desktop runtime that coordinates local inputs.
- [ ] A browser bridge that injects text and media into AI products.
- [ ] A mobile companion that streams speech, images, and files.
- [ ] A device mesh that discovers trusted local endpoints.
- [ ] A plugin system for websites, models, automation tools, and creative apps.
- [ ] A local-first agent runtime for personal workflows.

## Principles

- [ ] Local-first by default.
- [ ] Privacy-respecting by design.
- [ ] Composable rather than monolithic.
- [ ] Useful before it is grand.
- [ ] Friendly to contributors who want to add one adapter, one device, or one workflow.
- [ ] Honest about what is finished and what is still a frontier.

## Unfinished Questions

- [ ] What should AURA's plugin boundary look like?
- [ ] How should users grant trust between devices?
- [ ] Which model providers and local runtimes should be supported first?
- [ ] How should browser automation remain reliable as AI sites change?
- [ ] What should the mobile protocol become beyond upload and discovery?
- [ ] Should AURA define its own local message protocol or adopt an existing one?
